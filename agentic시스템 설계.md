# agentic시스템 설계

## 목표
- 사용자가 명확하고 강제된 입력을 제공하면, 에이전트는 **해석 없이** 코드베이스에 의도를 정확히 반영한다.
- 출력의 예측 가능성을 높이기 위해, 단계별로 필요한 정보만 전달한다.

## 핵심 원칙
- 모호함 최소화: 입력 스펙을 고정된 형식으로 제공(초기에는 overfit, 이후 필요 시 일반화).
- 설계가 대부분의 판단을 수행한다.
- 구현 단계는 설계를 **1:1로 수행**하며, 자체 판단을 하지 않는다.
- 단계마다 사람(사용자)이 검증하는 **게이트키퍼 구조**로 운영한다.

## 단계 구조 (사람이 main)
1) **조사(Research)**
   - 역할: 주어진 과제를 수행하기 위해 필요한 코드베이스 정보를 탐색·수집한다.
   - 전제: 블루프린트 문서는 두지 않는다. 매번 프로젝트 코드를 읽어 최신 상태를 반영한다.
   - 산출물: 설계 에이전트에게 전달할 **조사 자료**.

2) **사람 검증 (Research 결과 확인)**
   - 조사 결과의 정확성/누락/불확실성 확인.

3) **설계(Design)**
   - 입력: 사용자 과제 정의 + 조사 산출물
   - 역할: 실제로 어떤 파일을 수정/추가해야 하는지 **파일 단위 상세 설계 스펙**을 만든다.
   - 산출물: 구현 단계가 그대로 수행할 수 있는 설계 문서.

4) **사람 검증 (Design 결과 확인)**
   - 설계 결과의 정확성/누락/범위 확인.

5) **구현(Implement)**
   - 입력: 사용자 과제 정의 + 설계 산출물
   - 역할: 설계를 그대로 코드로 옮긴다(판단/재해석 금지).
   - 산출물: 코드 변경.

## 블루프린트 처리 방침
- 블루프린트는 별도 문서로 유지하지 않는다.
- 매 작업마다 **조사 단계에서 코드베이스를 읽고 청사진을 구성**한다.

## 철학
- 사람이 의도를 확정하고, 에이전트는 실행을 담당한다.
- 설계 단계에서 대부분의 결정이 내려지고, 구현은 “코드를 대신 써주는 단계”로 제한한다.

## 진행 순서 요약
- 조사 → 사람 검증 → 설계 → 사람 검증 → 구현
